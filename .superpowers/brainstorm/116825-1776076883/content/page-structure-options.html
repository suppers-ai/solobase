How should the use cases page be structured?

Three approaches for presenting 12 use cases across 3 groups. Click to select.

A

Single Long Page with Group Sections

One scrollable page. Hero at top, then 3 colored sections (Browser/WASM, Single Binary, Platform/Edge). Each use case is a card within its section — title, one-paragraph vision, expandable "how it works" detail.

Pros

  • Simple to build and maintain
  • Users see everything at a glance
  • Good for SEO — all content on one URL

Cons

  • 12 use cases on one page could feel long
  • Harder to link to a specific use case
B

Overview Page + Individual Use Case Pages

A hub page showing all 3 groups as visual cards with short descriptions. Each use case links to its own dedicated page with the full vision, architecture diagram, code snippets, and "how to get started." Like Vercel's or Supabase's use case pages.

Pros

  • Each use case gets room to breathe
  • Shareable individual URLs
  • Can deep-link from blog posts, docs, etc.

Cons

  • More pages to build and maintain (13 total)
  • Content might feel thin on some pages
C

Overview Page + Group Detail Pages

A hub page showing 3 groups. Each group links to its own page (e.g., /use-cases/browser, /use-cases/native, /use-cases/platform) where all use cases in that group are laid out with full detail. 4 pages total instead of 13.

Pros

  • Balanced — room for detail without 13 pages
  • Groups reinforce the "3 deployment models" story
  • Easier to maintain than individual pages

Cons

  • Can't link to a single use case directly
  • Group pages might feel like a wall of content

My recommendation: Option A — a single page. 12 use cases with short cards is digestible, anchor links solve deep-linking, and it's far less maintenance. You can always split into individual pages later if a use case needs its own marketing push.
